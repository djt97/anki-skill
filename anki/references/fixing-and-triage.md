## Preference discovery

Run this once after the first general health check when no saved `preferences` exist. Do not run it merely because a user made a specific unrelated request.

Select 3–5 of the user's own struggling cards that span different types when available:

- an isolated date or number;
- a citation, author, title, or naming card;
- an ambiguous or non-univocal prompt;
- a bundled-answer or list card;
- a genuinely hard concept whose construction may already be sound.

Never use an example from the skill files as though it were the user's card. Show only enough card content and review data to make the choice intelligible.

For each selected card, use the host's structured choice tool when available. The shape is:

> One card you struggle with  
> Front: “⟨their actual front⟩”  
> Back: “⟨their actual answer⟩”  
> How would you like cards like this handled?
>
> 1. Rewrite it around a different fact.
> 2. Keep the fact and add a memory hook or mnemonic.
> 3. Delete it.
> 4. Leave it as it is.
> 5. Something else.

Tailor the options to the card type. For an ambiguous card, for example, include “keep the target and make the wording univocal.” For a bundled citation, include “keep both facts but split author and year.”

The choice is preference evidence, not immediate approval to edit the shown card. After the interview, do not apply any change.

## Save preferences by card type

Record the user's answer by type so it generalizes. Suitable keys include the original profile:

```text
preferences:
  isolated_date: keep+mnemonic
  citation_naming: keep+split-author-year
  ambiguous: fix-wording
  hard_concept: leave
  costly_card: ask
```

Do not infer a preference from silence. When the answer applies only to one card, record no general default and ask again on future edge cases.

Show the proposed addition to `conventions.local.md` and obtain explicit approval before saving. Preserve all existing keys and user corrections.

## Choose a starting point

After calibration, stop and hand control back. Suitable choices are:

1. Review the costliest cards.
2. Work on a particular deck, tag, topic, or note type.
3. Work through the confirmed review-flag queue.
4. Review leeches or suspended cards.
5. Search for a particular subject.
6. Stop for now.

Do not select a queue for the user. Do not begin fixing the cards used in calibration unless the user chooses them.

## Build and approve fix batches

Use [references/card-writing.md](card-writing.md) to grade construction and choose a recommendation. For each mutation batch:

1. **Compose 1–4 items.** Use fewer for substantial edits; a major rewrite stands alone.
2. **Re-fetch full current data.** Do not rely on a truncated health-report field.
3. **Read the annotation field first** when one is configured.
4. **Check the source of truth** for every note.
5. **Inspect sibling cards** generated by the note.
6. **Apply the saved preference for that card type** unless the card is a genuine edge case or a costly card requiring a fresh choice.
7. **Show the complete batch and stop.** No changes have been made at this point.

For each card, display:

- card ID, note ID, deck, note type, tags, and sibling card IDs;
- front and back or all fields that matter;
- reps, lapses, lapse rate, interval, and cost score where available;
- the annotation field;
- T0–T3 construction grade;
- the recommended operation;
- exact before/after content or exact non-content operation;
- the effect on sibling cards and source files;
- **⚠ confirm** for facts not verified against the user's source.

Allowed recommendations are:

- **construction fix:** improve wording while keeping the target;
- **memory hook:** retain an atomic fact and add an optional encoding aid;
- **split:** create separate cards for separately recallable targets;
- **re-aim:** change what the card tests, only when a saved preference or explicit choice authorizes it;
- **unsuspend or suspend:** after a deliberate choice;
- **move:** change deck after showing the exact destination;
- **delete:** only after a deliberate choice and separate final confirmation;
- **keep:** make no change.

A split is a mutation to card content and potentially note creation. Preview every resulting note/card, tags, deck, and what will happen to the original. Do not silently delete or suspend the original after splitting.

## Handle costly cards

A costly card is in the worst `costly_pctile` percent by the custom cost score, default 5. It is not the same as an Anki leech.

For every costly card, surface the data and ask deliberately even when a broad saved preference would otherwise apply. Offer the full menu:

1. Encode it with a memory hook.
2. Split it.
3. Re-aim it.
4. Suspend it.
5. Delete it.
6. Keep it.
7. Rewrite the wording while keeping the target, when applicable.
8. Something else.

Do not choose on the user's behalf. A high cost score alone does not justify deletion or re-aiming.

## Triage suspended, leech, or review queues

When the user asks to triage a queue:

1. **Survey.** Find the matching cards and group them by deck and note type. Show exact counts.
2. **Review read-only batches of 5–10.** For each card, show front, back, tags, deck/type, lapse count, and the annotation field.
3. **Recommend, do not execute:**
   - **Unsuspend:** construction appears sound and the user wants it back in review.
   - **Rewrite + unsuspend:** wording is poor; preview the rewrite and the later unsuspension as separate explicit effects.
   - **Delete:** redundant or unwanted, but only the user decides.
   - **Keep suspended:** defer or retain outside review.
   - **Other:** split, move, memory hook, or re-aim when appropriate.
4. **Convert decisions into mutation batches of 1–4.** The larger read-only triage view is not blanket approval.
5. **Require separate deletion confirmation.** Show all cards generated by the note.

When a tag or flag meaning is not confirmed, describe it literally—such as “flag 5 cards”—rather than naming it “fix me” or “hard.”

## Apply and verify approved changes

After the user approves a displayed batch:

### Anki-native note

- Apply only the approved fields or operations through AnkiConnect.
- For a field edit, use the note ID and do not include untouched fields.
- Re-read the note and all affected sibling cards.
- Compare the stored content with the exact approved preview.

### Externally authored note

- Show the exact external file and diff before approval.
- Apply only the approved diff to the source file.
- Re-read the file and verify the marker and surrounding card source remain intact.
- Do not call `updateNoteFields` for that note.
- Remind the user to run their normal external-to-Anki sync.

### Tags

Tag changed notes only if the user approved the exact tag. If no change tag is part of their conventions, ask once or skip tagging. Do not invent `reviewed`, `rewritten`, or another label merely because an example uses it.

### Batch failures

`multi` is not atomic. Inspect every sub-result and re-read final state. If any part failed, state exactly which operations succeeded and which did not. Do not automatically retry a mutation after an ambiguous transport failure.

### Completion

Report:

- what was changed;
- what was left unchanged;
- verification results;
- whether an external re-sync or manual AnkiWeb sync is appropriate.

Then offer the next small batch or stop. Never continue applying edits from an earlier approval.
